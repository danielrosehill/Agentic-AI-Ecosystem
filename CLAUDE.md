# CLAUDE.md — Agentic AI Ecosystem

## Project objective

This repo is a **hand-curated taxonomy of the agentic AI ecosystem**. The goal is *not* an exhaustive project directory — it's a **classification map**. Example projects are collected as *evidence* that a category exists, then used in reverse to derive and refine the taxonomy itself.

Think of it as: "What are the real functional categories in agentic AI tooling right now?" — with a handful of representative projects pinned under each as proof.

## Working principles

1. **Granularity over consolidation.** Deep, specific subcategories are preferred over broad buckets. A category with one example (or even zero) is fine if it represents a distinct functional slot. Only deduplicate when two labels mean the *same* thing — not when they're merely related.

2. **Categories first, examples second.** When adding a project, the question to ask is *"what classification does this reveal or confirm?"* — not *"where can I dump this?"* If a project doesn't fit, that's a signal to create (or refine) a category, not to force-fit.

3. **Every segment and subcategory has a `description` field.** Even if blank (`""`). This is how sibling categories stay distinguishable and how future-us knows what belongs where.

4. **Alphabetical ordering** at every level of `segments[]` and `children[]`, unless a deliberate grouping justifies otherwise.

5. **Project entries are minimal**: `{ "name": "...", "url": "..." }`. No descriptions, no tags. The taxonomy carries the meaning.

6. **`OpenClaw`-style mistakes to avoid**: a specific project name should never become a subcategory. Subcategories are *functional slots*; projects live inside them.

## Data model

**Source of truth is the graph**: `graph/nodes.json` + `graph/edges.json`. `ecosystem.json` is a *regenerated* tree view kept around for `generate_readme.py` and downstream consumers.

### Graph schema

**Nodes** (`graph/nodes.json`)

```json
{
  "updated": "YYYY-MM-DD",
  "nodes": [
    {
      "id": "cat:frameworks/voice",
      "type": "Category",
      "label": "Voice",
      "path": ["Frameworks", "Voice"],
      "description": "Voice-first agent frameworks."
    },
    {
      "id": "proj:livekit-agents",
      "type": "Project",
      "label": "livekit/agents",
      "url": "https://github.com/livekit/agents"
    }
  ]
}
```

**Edges** (`graph/edges.json`)

```json
{
  "updated": "YYYY-MM-DD",
  "edges": [
    { "from": "cat:frameworks/voice", "to": "cat:frameworks", "type": "SUBCATEGORY_OF" },
    { "from": "proj:livekit-agents",   "to": "cat:frameworks/voice", "type": "CATEGORY_OF" }
  ]
}
```

Current edge types: `SUBCATEGORY_OF`, `CATEGORY_OF`. Future types (not yet populated): `IMPLEMENTS`, `BUILDS_ON`, `HOSTED_VERSION_OF`, `MAINTAINED_BY`, `RELATED_TO`.

**ID rules**
- Categories: `cat:<slug-path>` — e.g. `cat:frameworks/voice`
- Projects: `proj:<slug-name>`; on collision append `--<parent-slug>`
- Slug: lowercase, non-alphanumerics → `-`, trimmed

**Depth**: typically 2 levels of category (segment → subcat), occasionally 3. Avoid going deeper.

### Build pipeline

```
graph/nodes.json + graph/edges.json   (source of truth, hand-edited)
        │
        ├── scripts/build_tree.py       → ecosystem.json (derived)
        ├── scripts/generate_readme.py  → README.md     (derived, reads ecosystem.json)
        └── scripts/build_graph.py      (legacy: tree → graph, only for one-shot migrations)
```

After any graph edit, run `python3 scripts/build_tree.py` before `/update-readme`. Never hand-edit `ecosystem.json` — changes will be overwritten on the next build.

## Repo layout

- `graph/nodes.json` + `graph/edges.json` — source of truth (graph)
- `ecosystem.json` — derived tree view, regenerated by `build_tree.py`
- `scripts/build_tree.py` — graph → tree
- `scripts/build_graph.py` — tree → graph (legacy, used during migration)
- `scripts/generate_readme.py` — regenerates README from the JSON tree
- `.claude/commands/` — slash commands for repo workflows
- `README.md` — generated

## Slash commands

- `/add` — ingest a new project (with classification reasoning)
- `/organise` — reorganise structure
- `/update-readme` — regenerate README from ecosystem.json
- `/taxonomy-review` — dedup + description audit pass
- `/workflow` — overall workflow guide

## Workflow for ingesting a new project

1. Read the project (URL, description, README if available).
2. Identify the **functional slot** it occupies. Is it a framework? A runtime? A harness? A specific kind of tool?
3. Check whether that slot already exists in `ecosystem.json`:
   - **Yes** → append to `examples[]`.
   - **No, but close** → consider whether a new subcategory under an existing segment is warranted.
   - **No, genuinely new** → create the segment/subcategory *with a description* explaining what belongs there.
4. If the project reveals a *distinction* that the current taxonomy flattens, split the existing category.
5. Bump `updated`.
6. Regenerate README.

## Do not

- Add descriptions to individual projects.
- Write prose commentary in `ecosystem.json`.
- Force-fit projects into the wrong category because no better slot exists — create the slot.
- Flatten granular categories "to clean up" — granularity is the point.
- Leak private/client info; this repo is public.
